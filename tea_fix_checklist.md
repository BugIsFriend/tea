# Tea 框架修复 Checklist

> 目的：把 `tea_framework_review.md` 中的分析结论，拆成可执行、可复查、可勾选的整改清单。
>
> 使用方式：每完成一项就打勾，并记录验证结果；不要一次性改完整个框架后再统一测试。

---

## 一、执行原则

- [ ] 每次只改一个模块或一个明确问题点
- [ ] 每改完一个问题点，立即做一次最小验证
- [ ] 优先修基础层正确性，再修 UI，再做重构
- [ ] 不把“顺手优化”混进当前修复项
- [ ] 修改前先确认问题是否仍然存在，避免修历史问题

---

## 二、P0：先修基础正确性问题

### 1. `assets/tea/storage.ts`

#### 目标

修复本地存储的结构错误、过期逻辑错误和输入对象副作用。

#### 修改项

- [ ] 检查 `decode()` 是否返回完整 `StorageValue<T>` 结构
- [ ] 修复 `get()` 对 `decode()` 返回值的使用方式
- [ ] 验证 `expire` 到期后会返回 `null` 并自动删除 key
- [ ] 改掉 `set()` 中对入参 `data` 的 `delete` 副作用
- [ ] 统一 `set/get/remove` 对 `id` 拼接规则的支持方式

#### 验证项

- [ ] 普通数据 `set -> get` 能正确读回
- [ ] 带过期时间的数据到期后返回 `null`
- [ ] 同一 key + 不同 id 的数据互不影响
- [ ] 传入 `set()` 的原始对象不会被改写

#### 建议记录

- 记录测试 key
- 记录过期前/过期后行为

---

### 2. `assets/tea/emitter.ts`

#### 目标

修复监听误删、延迟消息参数错位、优先级排序无效等问题。

#### 修改项

- [ ] 修复 `checkEmmit()` 的有效性判断逻辑
- [ ] 修复 `delay().emit()` 参数转发方式
- [ ] 检查 `on()` 中优先级排序是否真正生效
- [ ] 补充 `off()` 对 once 队列的清理
- [ ] 统一拼写与命名（如 `checkEmmit` / `emmiter`）

#### 验证项

- [ ] `on -> emit` 能正常触发
- [ ] `once -> emit` 只触发一次
- [ ] `off({ id, context })` 能正确移除指定监听
- [ ] `off({ context })` 能清理该对象所有监听
- [ ] `delay().emit()` 与直接 `emit()` 参数表现一致
- [ ] 多个不同优先级监听按预期顺序触发

#### 建议记录

- 记录测试消息名
- 记录不同优先级下的触发顺序

---

### 3. `assets/tea/unit.ts`

#### 目标

修复 `gain()` 初始化错误对象的问题。

#### 修改项

- [ ] 确认新增组件后调用的是新组件本身的初始化逻辑
- [ ] 确认已存在组件不会重复初始化
- [ ] 如果 `Component` 不一定有 `init`，加存在性判断

#### 验证项

- [ ] 首次 `gain()` 创建组件时能执行初始化
- [ ] 第二次 `gain()` 只返回已存在组件，不重复初始化
- [ ] 对普通无 `init` 组件也不会报错

---

### 4. `assets/tea/fsm/fsm.ts`

#### 目标

修复 `owner` 未赋值、初始状态不安全等问题。

#### 修改项

- [ ] 在初始化时设置 `this.owner = this.node`
- [ ] 校验 `curState` 为空时的处理逻辑
- [ ] 避免对空状态名执行 `addComponent('')`
- [ ] 切换状态前后增加存在性保护

#### 验证项

- [ ] 初始状态能正确进入
- [ ] `enter/execute/exit` 收到的 owner 是当前 node
- [ ] 切换到不存在状态时有明确反馈，不会静默异常
- [ ] 相同状态重复切换时不会重复执行切换流程

## 三、P1：修 UI 主链路问题

### 5. `assets/tea/ui.ts`

#### 目标

修复弹窗返回值、默认参数污染和堆栈行为不稳定问题。

#### 修改项

- [ ] 修复 `show()` 异步 load 分支缺少 `return`
- [ ] 避免 `defaultParam` 被 `Object.assign` 直接污染
- [ ] 明确重复 `show` 同一个 view 的行为
- [ ] 检查关闭当前 view 后顶部 view 的恢复逻辑
- [ ] 检查背景层 siblingIndex 与 active 状态切换是否稳定

#### 验证项

- [ ] `await ui.load(...).show()` 能返回有效 `View`
- [ ] 连续打开多个弹窗，背景参数不会串用
- [ ] 关闭顶部弹窗后，下层弹窗能正常恢复显示
- [ ] 同一个 view 重复 show 时行为可预期

- [ ] 为 `close()` 增加无动画兜底完成态
- [ ] 检查 `appendShowedCb` / `appendClosedCb` 是否需要去重或一次性消费
- [ ] 确认 `EventHandler.emit` 参数传递约定是否统一

#### 验证项

- [ ] 配置合法动画时能正常进入完成态
- [ ] 无动画配置时也不会卡在 `Opening/Closing`
- [ ] 打开/关闭回调数量符合预期，不重复触发

- [ ] 检查 `touch` / `touchClose` / `intercept` 的组合逻辑

## 八、复查建议

- [ ] 每完成一个阶段，回看 `tea_framework_review.md`
- [ ] 每完成一个模块，补一条“是否还需要重构”的判断
- [ ] 如果某问题已被当前代码自然消除，及时在 checklist 标注“已确认无需处理”
- [ ] 若发现新的高优先级 bug，先插入 P0/P1，再继续原计划

---

## 九、一句话执行策略

> 不要一口气重构整个 `tea`，而是先把底层 bug 修到“行为可信”，再去做 UI 稳定化，最后再整理单例、装饰器和复杂组件结构。
